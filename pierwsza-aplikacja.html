Faza 1 — Planowanie
1
Start

Wybór tematu, platformy i składu zespołu

~1–2 godziny Przed Lab 4 Product Lead

Zanim napiszesz pierwszą linię kodu, musisz wiedzieć: co budujesz, dla kogo i w jakiej technologii. Poniższe decyzje podejmij razem z zespołem.

  • Ustal skład zespołu (dokładnie 3 osoby) i przypisz role: Product Lead & UX Frontend Developer Backend & DevOps
  • Wybierz temat aplikacji — musi ona realnie wykorzystywać możliwości smartfona (np. GPS, aparat, sensory), a nie być prostą wersją aplikacji webowej.
  • Wybierz platformę: Android (Kotlin + Jetpack Compose), Flutter lub inną uzgodnioną z prowadzącym.
  • Zidentyfikuj grupę docelową użytkowników i ich główny problem, który rozwiązujesz.
  • Zapisz listę co najmniej 5 kluczowych funkcjonalności (MVP) i odrębną listę funkcji dodatkowych (nice-to-have).
Wskazówka Dobry temat to taki, w którym aplikacja bez smartfona (czyli np. jako strona internetowa na komputerze) straciłaby sens. Przykład: aplikacja do skanowania paragonów z aparatu — bez aparatu byłaby bezużyteczna.
2
Wymagany dokument

Dokument wymagań funkcjonalnych (SRS / User Stories)

~3–5 godzin Lab 4 — termin zgłoszenia Product Lead

SRS (Software Requirements Specification) to podstawowy dokument każdego projektu programistycznego. Opisuje co system ma robić, nie jak. W projekcie PAM zastępują go User Stories z kryteriami akceptacji.

Struktura dokumentu SRS / User Stories:

  • 1
    Cel projektu — 1 zdanie opisujące problem i rozwiązanie.
  • 2
    Persony użytkowników — 2–3 opisy typowego użytkownika.
  • 3
    User Stories — min. 15, każda w formacie: Jako [rola] chcę [funkcja] aby [korzyść].
  • 4
    Kryteria akceptacji dla każdej historyjki (kiedy uznajemy ją za "zrobioną").
  • 5
    Lista ekranów — tytuł + krótki opis każdego ekranu (min. 5).
  • 6
    Wymagania niefunkcjonalne — wydajność, bezpieczeństwo, dostępność.
Przykład User Story z kryteriami akceptacji
US-07: Jako zalogowany użytkownik chcę dodać nowe zadanie do listy,
        aby móc śledzić swoje obowiązki.

Kryteria akceptacji:
  ✓ Formularz zawiera pole: tytuł (wymagane), opis, termin, priorytet
  ✓ Po zapisaniu zadanie pojawia się natychmiast na liście
  ✓ Walidacja: tytuł nie może być pusty (komunikat błędu)
  ✓ Zadanie jest persystowane lokalnie (działa offline)
  • Napisz min. 15 User Stories w Google Docs / Markdown / Notion.
  • Każda User Story musi mieć min. 3 kryteria akceptacji.
  • Uzasadnij dlaczego aplikacja wymaga smartfona (jakie natywne funkcje i dlaczego).
  • Opisz zakres MVP (co musi działać na prezentacji) vs. funkcje dodatkowe.
Ważne — termin Dokument SRS / User Stories musisz zaprezentować prowadzącemu na Laboratorium nr 4. Bez zaakceptowanego tematu nie możesz kontynuować projektu.

GitHub

Konfiguracja repozytorium GitHub

~1 godzina Dzień 1 projektu DevOps

Repozytorium GitHub to centralne miejsce pracy całego zespołu. Prawidłowa konfiguracja od początku uchroni Was przed konfliktami i problemami z historią commitów.

  • 1
    Zaloguj się na github.com i kliknij New repository. Ustaw widoczność na Private lub Public (uzgodnij z prowadzącym). Zaznacz "Add README".
  • 2
    Dodaj współpracowników: Settings → Collaborators → Add people i dodaj pozostałych członków zespołu po nazwie użytkownika.
  • 3
    Utwórz branch protection rule dla gałęzi main: Settings → Branches → Add rule. Zaznacz "Require a pull request before merging" i "Require status checks to pass before merging".
  • 4
    Utwórz dwa milestone'y: Issues → Milestones → New milestone: Sprint 1 — Lab 4–6 (MVP) i Sprint 2 — Lab 7–8 (finalny).
  • 5
    Utwórz projekt Kanban w GitHub Projects: Projects → New project (Board view) z kolumnami: Backlog / In Progress / In Review / Done.
  • 6
    Utwórz pierwsze Issues na podstawie User Stories (min. 10 na start). Przypisz je do odpowiednich osób i milestone'ów.
Zalecana struktura branchy (Git Flow)
main          ← tylko stabilny, przetestowany kod
develop       ← integracja zmian od wszystkich developerów
feature/xxx   ← nowa funkcja (np. feature/login-screen)
fix/xxx       ← poprawka błędu
release/x.y   ← przygotowanie wydania
  • Repozytorium założone, wszyscy członkowie zespołu mają dostęp.
  • Branch protection na main — wymagany PR i review przed merge.
  • 2 milestone'y (Sprint 1 i Sprint 2) skonfigurowane.
  • Projektu zaliczony wymaga min. 20 zamkniętych Issues.

Architektura

Szkielet projektu i architektura MVVM

~2–3 godziny Po Lab 4 Frontend Dev

Architektura MVVM (Model-View-ViewModel) to standard w aplikacjach Android. Rozdziela logikę biznesową od interfejsu użytkownika — kod jest testowalny i łatwy w utrzymaniu.

  • 1
    W Android Studio kliknij New Project → Empty Activity (Compose). Ustaw Package name (np. pl.prz.pam.taskmaster), minSdk: 26, targetSdk: 34.
  • 2
    Utwórz strukturę pakietów odpowiadającą warstwom MVVM:
Zalecana struktura pakietów
com.example.app/
├── data/
│   ├── local/          ← Room DAO, Database, entities
│   ├── remote/         ← Retrofit API, DTOs
│   └── repository/     ← implementacje repozytoriów
├── domain/
│   ├── model/          ← modele domenowe
│   └── usecase/        ← logika biznesowa
├── ui/
│   ├── screens/        ← Composables dla każdego ekranu
│   │   ├── home/
│   │   ├── detail/
│   │   └── settings/
│   ├── components/     ← reużywalne komponenty UI
│   ├── navigation/     ← NavGraph
│   └── theme/          ← kolory, typografia, kształty
└── di/                 ← Hilt / Koin modules
  • 3
    Dodaj zależności do build.gradle.kts: Hilt (DI), Navigation Compose, Room, Retrofit, Coil (obrazy), ViewModel Compose.
  • 4
    Stwórz pierwszy commit na gałęzi feature/project-setup i otwórz Pull Request do develop.
Przykładowy ViewModel (Kotlin + Compose)
@HiltViewModel
class TaskViewModel @Inject constructor(
    private val taskRepository: TaskRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(TaskUiState())
    val uiState: StateFlow<TaskUiState> = _uiState.asStateFlow()

    fun loadTasks() {
        viewModelScope.launch {
            taskRepository.getTasks().collect { tasks ->
                _uiState.update { it.copy(tasks = tasks, isLoading = false) }
            }
        }
    }
}

Nawigacja

Implementacja nawigacji i ekranów

~6–10 godzin Sprint 1 Frontend Dev

Nawigacja to "szkielet" aplikacji łączący wszystkie ekrany. W Jetpack Compose używamy Navigation Compose — deklaratywnego systemu opartego o NavController.

  • 1
    Zdefiniuj sealed class lub enum ze wszystkimi trasami (routes) nawigacji.
  • 2
    Stwórz AppNavGraph — composable z NavHost i wszystkimi composable { } destinacjami.
  • 3
    Zaimplementuj każdy ekran jako osobny Composable w osobnym pliku. Ekran główny, lista, szczegóły, formularz, ustawienia.
  • 4
    Dodaj bottom navigation bar lub navigation drawer dla głównych sekcji.
  • 5
    Obsłuż przycisk Wstecz (Back Handler) i poprawne zarządzanie back stackiem.
Szkielet NavGraph (Jetpack Compose)
@Composable
fun AppNavGraph(navController: NavHostController) {
    NavHost(navController, startDestination = Screen.Home.route) {
        composable(Screen.Home.route) {
            HomeScreen(onNavigateToDetail = { id ->
                navController.navigate(Screen.Detail.createRoute(id))
            })
        }
        composable(
            Screen.Detail.route,
            arguments = listOf(navArgument("id") { type = NavType.LongType })
        ) { backStackEntry ->
            val id = backStackEntry.arguments?.getLong("id") ?: return@composable
            DetailScreen(itemId = id, onBack = { navController.popBackStack() })
        }
    }
}
  • Min. 5 ekranów z działającą nawigacją między nimi.
  • Stany UI: Loading / Content / Empty / Error — każdy ekran obsługuje te stany.
  • Obsługa stanu offline (wyświetlenie stosownego komunikatu lub danych z cache'u).

Dane lokalne

Lokalna baza danych (Room)

~3–5 godzin Sprint 1 Frontend Dev

Room to ORM (Object-Relational Mapper) dla SQLite na Androidzie. Umożliwia przechowywanie danych lokalnie — aplikacja działa offline i jest szybsza.

  • 1
    Dodaj zależność room-runtime, room-ktx i room-compiler do build.gradle.kts.
  • 2
    Stwórz klasy @Entity dla każdej tabeli (np. TaskEntity).
  • 3
    Stwórz interfejsy @Dao z metodami @Insert, @Update, @Delete, @Query.
  • 4
    Stwórz klasę @Database rozszerzającą RoomDatabase. Użyj Room.databaseBuilder() w module Hilt.
  • 5
    Zaimplementuj klasę Repository łączącą Room z resztą aplikacji (ViewModel nie powinien znać Room bezpośrednio).
Przykładowa encja i DAO
@Entity(tableName = "tasks")
data class TaskEntity(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val title: String,
    val description: String,
    val isDone: Boolean = false,
    val createdAt: Long = System.currentTimeMillis()
)

@Dao
interface TaskDao {
    @Query("SELECT * FROM tasks ORDER BY createdAt DESC")
    fun getAllTasks(): Flow<List<TaskEntity>>

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertTask(task: TaskEntity): Long

    @Delete
    suspend fun deleteTask(task: TaskEntity)
}

Backend i API

Backend, uwierzytelnianie i integracja z API

~8–15 godzin Sprint 1–2 DevOps Frontend Dev

Backend zapewnia serwer, bazę danych i API, z którym komunikuje się aplikacja mobilna. Możesz zbudować własny backend (Node.js, Spring Boot, Django) lub skorzystać z gotowej usługi (Firebase, Supabase).

Opcja backenduZaletyWady
Firebase (Google) Szybki start, brak konfiguracji serwera, auth gotowy Uzależnienie od usługi, ograniczone darmowe limity
Node.js + Express Pełna kontrola, łatwy do nauki Trzeba skonfigurować hosting
Supabase Open-source Firebase, PostgreSQL, REST auto-generated Nowa platforma, mniejsza dokumentacja
  • 1
    (Backend) Zaimplementuj min. 5 endpointów REST: rejestracja, logowanie, oraz min. 3 operacje CRUD na głównych danych.
  • 2
    (Backend) Dodaj uwierzytelnianie JWT lub OAuth. Hashuj hasła (bcrypt). Chroń wszystkie prywatne endpointy middleware'em auth.
  • 3
    (Backend) Udokumentuj API przy użyciu Swagger / OpenAPI.
  • 4
    (Frontend) Dodaj Retrofit + OkHttp do projektu Android. Stwórz interfejs API i klas DTOs.
  • 5
    (Frontend) Przechowuj tokeny bezpiecznie w EncryptedSharedPreferences lub Keystore, nigdy w zwykłych SharedPreferences.
  • 6
    (Frontend) Obsłuż błędy sieciowe: brak internetu, timeout, 401 (odświeżenie tokenu), 5xx (komunikat dla użytkownika).
  • 7
    (DevOps) Wdróż backend na serwerze publicznym (np. Railway, Render, Heroku, VPS). API musi być dostępne z zewnątrz.
  • Min. 5 endpointów API (w tym auth: rejestracja + logowanie).
  • Działający przepływ end-to-end: logowanie → operacja domenowa → wynik.
  • Bezpieczne przechowywanie tokenów (nie w plaintext).

Testy

Testy jednostkowe i integracyjne

~4–6 godzin Sprint 1–2, równolegle z implementacją Frontend Dev DevOps

Testy zapewniają, że kod działa poprawnie i że przyszłe zmiany niczego nie psują. Wymagane są testy jednostkowe frontendu (≥60% pokrycia logiki) oraz testy integracyjne API (min. 5 scenariuszy).

  • 1
    (Android) Piszesz testy do katalogu test/ (testy jednostkowe JVM). Użyj JUnit4 + MockK/Mockito do mockowania zależności. Testuj ViewModele, UseCases, Repozytoria.
  • 2
    Uruchom testy: Run → All Tests w Android Studio lub ./gradlew test w terminalu.
  • 3
    Sprawdź pokrycie kodu: Run → Run with Coverage. Cel: ≥60% pokrycia klas logiki.
  • 4
    (Backend) Napisz min. 5 testów integracyjnych API (np. Supertest dla Node.js, pytest dla Python). Testuj: rejestrację, logowanie, CRUD, obsługę błędów.
Przykładowy test jednostkowy ViewModelu (Kotlin)
@ExperimentalCoroutinesApi
class TaskViewModelTest {
    @get:Rule val mainCoroutineRule = MainCoroutineRule()

    private val mockRepository: TaskRepository = mockk()
    private lateinit var viewModel: TaskViewModel

    @Before
    fun setup() {
        every { mockRepository.getTasks() } returns flowOf(listOf(fakeTask))
        viewModel = TaskViewModel(mockRepository)
    }

    @Test
    fun `loadTasks emits tasks in uiState`() = runTest {
        viewModel.loadTasks()
        val state = viewModel.uiState.first()
        assertEquals(1, state.tasks.size)
        assertEquals("Test Task", state.tasks[0].title)
    }
}
  • Min. 10 testów jednostkowych na frontendzie (ViewModel / UseCase / Repository).
  • Min. 5 testów integracyjnych backendu API.
  • Pokrycie kodu logiki ≥60%.

CI/CD

Konfiguracja CI/CD z GitHub Actions

~2–3 godziny Sprint 1 — jak najwcześniej DevOps

CI/CD (Continuous Integration / Continuous Delivery) automatycznie uruchamia build i testy przy każdym Pull Requeście. Dzięki temu błędy są wykrywane natychmiast.

  • 1
    W repozytorium utwórz plik: .github/workflows/android-ci.yml
  • 2
    Skonfiguruj workflow tak, aby przy każdym PR do develop lub main uruchamiał: build projektu i testy jednostkowe.
  • 3
    Dodaj badge CI do pliku README.md.
.github/workflows/android-ci.yml
name: Android CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Build with Gradle
        run: ./gradlew assembleDebug --no-daemon

      - name: Run unit tests
        run: ./gradlew test --no-daemon
  • GitHub Actions workflow uruchamia się automatycznie przy każdym PR.
  • Build i testy muszą przejść przed merge do main.

Build

Podpisany build APK / AAB

~1–2 godziny Sprint 2 Frontend Dev

Każda aplikacja Android musi być podpisana cyfrowo przed dystrybucją. Google Play wymaga formatu AAB (Android App Bundle). Keystore to Twój "podpis deweloperski" — przechowuj go bezpiecznie i nigdy nie wrzucaj do repozytorium!

  • 1
    W Android Studio: Build → Generate Signed Bundle / APK. Wybierz Android App Bundle.
  • 2
    Kliknij Create new keystore. Wypełnij dane (alias, hasło, dane organizacji). Zapisz plik .jks w bezpiecznym miejscu poza repozytorium.
  • 3
    Skonfiguruj podpisywanie automatyczne w build.gradle.kts (hasła z zmiennych środowiskowych lub local.properties).
  • 4
    Ustaw versionCode i versionName w build.gradle.kts (np. versionCode 1, versionName "1.0.0").
Bezpieczeństwo keystore Plik .jks i hasła dodaj do .gitignore. Jeśli stracisz keystore, nie będziesz mógł aktualizować aplikacji w Google Play! Trzymaj kopię zapasową w bezpiecznym miejscu (np. zaszyfrowany plik w chmurze).
  • Wygenerowany podpisany AAB lub APK gotowy do dystrybucji.
  • Keystore nie jest w repozytorium (.gitignore).

Google Play

Publikacja w Google Play Store

~2–3 godziny Sprint 2 — przed Lab 8 Product Lead DevOps

Publikacja w Google Play to finał projektu. Wymaga jednorazowej opłaty rejestracyjnej (25 USD) i weryfikacji konta. Możesz opublikować na kanale testowym (Internal Testing) — nie wymaga pełnego przeglądu Google.

  • 1
    Zarejestruj konto Google Play Developer na play.google.com/console (jednorazowa opłata 25 USD). Weryfikacja może potrwać do 48h.
  • 2
    Utwórz nową aplikację w Play Console: Create app. Wybierz kategorię, język, typ (free/paid).
