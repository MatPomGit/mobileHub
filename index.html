Roadmapa — 6 etapów

Każde ćwiczenie buduje na poprzednim. Nie pomijaj etapów — każda koncepcja pojawia się ponownie w kolejnym ćwiczeniu.

🔤

Ćwiczenie 1 — Podstawy Kotlina i pierwsze @Composable

Lab 1

Instalacja Android Studio, konfiguracja emulatora (AVD), struktura projektu. Pierwsze funkcje @Composable — deklaratywne opisywanie UI zamiast imperatywnego zarządzania widokami. Podstawy składni Kotlin.

val / var fun / lambda @Composable remember { } mutableStateOf Column / Row / Box
Dlaczego Compose, nie XML? Stary system XML wymaga ręcznego synchronizowania widoków ze stanem (findViewById, setText…). Compose opisuje jak ekran wygląda przy danym stanie — gdy stan się zmieni, framework sam przerysowuje co trzeba. Mniej kodu, mniej błędów.
🗺️

Ćwiczenie 2 — Nawigacja i architektura MVVM

Lab 2

Wieloekranowe aplikacje z Navigation Compose. Wzorzec MVVM jako oficjalna architektura rekomendowana przez Google — separuje logikę (ViewModel) od wyglądu (Compose). StateFlow reaktywnie propaguje stan do UI.

ViewModel MutableStateFlow collectAsStateWithLifecycle NavController NavHost UiState data class
Dlaczego ViewModel? ViewModel przeżywa rotację ekranu — stan aplikacji nie resetuje się gdy obrócisz telefon. Dzięki separacji możesz testować logikę bez uruchamiania emulatora.
🗄️

Ćwiczenie 3 — Room Database i Coroutines

Lab 3

Dane muszą przeżyć zamknięcie aplikacji. Room to warstwa abstrakcji nad SQLite — definiujesz tabele przez @Entity, zapytania przez @Dao. Kotlin Coroutines (suspend fun) i Flow pozwalają pisać asynchroniczny kod bez blokowania UI.

@Entity @Dao + @Query @Database Flow<List<T>> suspend fun Migracje schematu
Dlaczego Flow, nie LiveData? Flow jest częścią Kotlin Coroutines — działa na dowolnej platformie (nie tylko Android), obsługuje backpressure i ma bogatszy zestaw operatorów. LiveData był Androidowym obejściem problemu, Flow jest właściwym rozwiązaniem.
🌐

Ćwiczenie 4 — REST API, Retrofit i Coil

Lab 4

Komunikacja z serwerem przez HTTP. Retrofit + OkHttp = kompletny stos sieciowy. Gson parsuje JSON na Kotlin data classes. Sealed class Result<T> obsługuje błędy sieciowe bezpiecznie. Coil ładuje i cachuje obrazy z URL.

Retrofit @GET/@POST OkHttp Interceptor Gson @SerializedName sealed Result<T> AsyncImage Offline-First
Dlaczego sealed class Result, nie try-catch? try-catch w Kotlinie jest opcjonalne — kompilator nie wymusi jego użycia. Sealed class Result<T> jest zwracany jako typ — when(result) musi obsłużyć Success, Error i Loading. Pominięcie = błąd kompilacji, nie crash w runtime.
💉

Ćwiczenie 5 — Hilt (DI) i testy jednostkowe

Lab 5

Dependency Injection eliminuje ręczne tworzenie i przekazywanie zależności. @HiltViewModel zastępuje fabryki ViewModel. Testy jednostkowe (JUnit + MockK) weryfikują logikę ViewModel bez emulatora. Turbine testuje Flow deterministycznie.

@HiltAndroidApp @HiltViewModel @Inject @Module @Binds JUnit + MockK Turbine
Dlaczego DI? Bez DI klasa A tworzy klasę B wewnętrznie — niemożliwe do podmienienia w testach. Z DI klasa A dostaje B z zewnątrz — w testach wstrzykujesz FakeB lub MockB. Kod jest testowalny i elastyczny.
🚀

Ćwiczenie 6 — WorkManager, DataStore i publikacja

Lab 6

WorkManager wykonuje zadania w tle gwarantując ich ukończenie nawet po restarcie telefonu. DataStore zastępuje SharedPreferences reaktywnym magazynem danych. Na końcu: podpisanie APK, konfiguracja ProGuard i Google Play Console.

WorkManager CoroutineWorker DataStore Preferences Proto DataStore ProGuard Play Console
Dlaczego WorkManager, nie coroutine w tle? Coroutine w viewModelScope jest anulowana gdy użytkownik zamknie aplikację. WorkManager gwarantuje uruchomienie zadania nawet po restarcie systemu, respektuje ograniczenia baterii i sieci.

TaskListScreen.kt 
@Composable
fun TaskListScreen(
    viewModel: TaskViewModel = viewModel()
) {
    val uiState by viewModel.uiState
        .collectAsStateWithLifecycle()

    Scaffold(
        topBar = {
            TopAppBar(
                title = { Text("Zadania") }
            )
        }
    ) { padding ->
        when {
            uiState.isLoading ->
                CircularProgressIndicator()
            uiState.error != null ->
                ErrorScreen(uiState.error!!)
            else ->
                LazyColumn(
                    contentPadding = padding
                ) {
                    items(
                        uiState.tasks,
                        key = { it.id }
                    ) { task ->
                        TaskItem(task = task)
                    }
                }
        }
    }
}
Linia koduCo oznacza i dlaczego
@Composable Deklaratywny UI — ta funkcja opisuje jak ekran wygląda przy danym stanie. Compose wywołuje ją ponownie gdy stan się zmieni.
viewModel() Automatyczne tworzenie — Compose tworzy lub pobiera istniejący ViewModel z zakresu (reuse przy rotacji ekranu).
collectAsStateWithLifecycle() Lifecycle-aware — subskrybuje StateFlow tylko gdy aktywność jest na pierwszym planie. Oszczędza baterię.
when { isLoading / error / else } Trzy stany UI — Loading (spinner), Error (komunikat), Success (lista). Wzorzec pokrywa wszystkie przypadki bez else-if spaghetti.
key = { it.id } Stabilna tożsamość — LazyColumn wie który element się zmienił i animuje tylko jego wiersz, nie całą listę.
TaskItem.kt 
@Composable
fun TaskItem(
    task: Task,
    onToggle: () -> Unit,
    onDelete: () -> Unit,
) {
    Card(
        modifier = Modifier
            .fillMaxWidth()
            .padding(8.dp)
            .animateContentSize()
    ) {
        Row(
            verticalAlignment =
                Alignment.CenterVertically,
            modifier = Modifier.padding(16.dp)
        ) {
            Checkbox(
                checked = task.isDone,
                onCheckedChange = { onToggle() }
            )
            Text(
                text = task.title,
                modifier = Modifier.weight(1f),
                textDecoration = if (task.isDone)
                    TextDecoration.LineThrough
                else TextDecoration.None
            )
        }
    }
}
Linia koduCo oznacza i dlaczego
onToggle: () -> Unit Event hoisting — TaskItem nie zmienia stanu sam. Zdarzenia "wyciągamy" do rodzica/ViewModel. Item jest stateless i łatwiejszy do testowania.
.animateContentSize() Automatyczna animacja — gdy Card zmienia rozmiar (np. po rozwinięciu), Compose interpoluje zmianę. Zero dodatkowego kodu.
.weight(1f) Flex w Row — Text zajmuje całe pozostałe miejsce po Checkbox. Responsywne na każdym rozmiarze ekranu.
if (task.isDone) ... else ... if jako wyrażenie — w Kotlin if zwraca wartość. Nie ma ternary operatora — if/else jest zwięźlejsze i czytelniejsze.
TaskViewModel.kt 
class TaskViewModel(
    private val repo: TaskRepository
) : ViewModel() {

    private val _uiState =
        MutableStateFlow(TaskUiState())

    val uiState =
        _uiState.asStateFlow()

    init {
        viewModelScope.launch {
            repo.getAllTasks()
                .collect { tasks ->
                    _uiState.update {
                        it.copy(tasks = tasks)
                    }
                }
        }
    }

    fun addTask(text: String) {
        if (text.isBlank()) return
        viewModelScope.launch {
            repo.save(Task(title = text))
        }
    }
}
Linia koduCo oznacza i dlaczego
: ViewModel() Przeżywa rotację — Android tworzy nową Activity/Fragment, ale ten sam ViewModel. Stan aplikacji nie ginie po obróceniu telefonu.
MutableStateFlow vs asStateFlow() Enkapsulacja — prywatne MutableStateFlow pisze ViewModel. UI dostaje read-only StateFlow. Modyfikacja tylko przez publiczne metody.
viewModelScope.launch { } Auto-cancel — coroutine anulowana automatycznie gdy ViewModel jest niszczony. Zero memory leaks, zero ręcznego zarządzania cyklem życia.
repo.getAllTasks().collect { } Reaktywność — Room Flow emituje nową listę przy każdej zmianie w bazie. ViewModel aktualizuje uiState → Compose przerysowuje ekran automatycznie.
it.copy(tasks = tasks) Niezmienialny stan — data class copy() tworzy nowy obiekt z jednym zmienionym polem. Pozostałe pola (isLoading, error) są niezmienione.
TaskUiState.kt 
// Jeden obiekt = pełny opis stanu ekranu
data class TaskUiState(
    val tasks:     List<Task>  = emptyList(),
    val isLoading: Boolean       = false,
    val error:     String?       = null,
)

// sealed class: kompilator zna WSZYSTKIE podklasy
// → when musi je wszystkie obsłużyć
sealed class Filter {
    data object All    : Filter()
    data object Active : Filter()
    data object Done   : Filter()
}

// when jest wyczerpujące — pominięcie = błąd
fun filtered(tasks: List<Task>, f: Filter) =
    when (f) {
        Filter.All    -> tasks
        Filter.Active -> tasks.filter { !it.isDone }
        Filter.Done   -> tasks.filter { it.isDone }
    }
Linia koduCo oznacza i dlaczego
data class TaskUiState Jeden source of truth — cały stan ekranu w jednym obiekcie. Compose subskrybuje go raz. Łatwiejsze debugowanie — logujesz jeden obiekt.
error: String? = null Nullable = opcjonalne — null oznacza "brak błędu". Nie trzeba oddzielnego boolean hasError. Kompilator wymaga sprawdzenia null przed użyciem.
sealed class Filter Zamknięta hierarchia — tylko 3 możliwe filtry zdefiniowane w tym pliku. Kompilator może sprawdzić czy when obsługuje każdy z nich.
data object All Kotlin 1.9+ — data object ma poprawną equals/hashCode/toString. Lepsze niż zwykłe object dla wartości w sealed class.
TaskEntity.kt 
// @Entity → Room tworzy tabelę w SQLite
@Entity(tableName = "tasks")
data class TaskEntity(

    @PrimaryKey
    val id: String = UUID.randomUUID()
                        .toString(),

    @ColumnInfo(name = "task_title")
    val title: String,

    @ColumnInfo(
        name = "is_done",
        defaultValue = "0"
    )
    val isDone: Boolean = false,

    val createdAt: Long =
        System.currentTimeMillis(),
)
Linia koduCo oznacza i dlaczego
@Entity(tableName = "tasks") Definicja tabeli — Room generuje CREATE TABLE tasks(...) automatycznie. Zmiana klasy bez migracji = crash przy uruchomieniu.
@PrimaryKey Unikalny klucz — każdy rekord musi mieć unikalny id. UUID jako String = bezpieczne przy synchronizacji z serwerem (nie kolizje z auto-increment).
@ColumnInfo(name = "task_title") Nazwa kolumny — kolumna w SQLite nazywa się "task_title", pole Kotlin "title". Jak @SerializedName dla JSON. Ważne przy migracji.
defaultValue = "0" Migracja bezpieczna — gdy dodajesz nową kolumnę do istniejącej tabeli, stare rekordy dostają tę wartość. Bez defaultValue migracja failuje.
TaskDao.kt 
// @Dao: interfejs zapytań SQL
// Room generuje IMPLEMENTACJĘ za nas!
@Dao
interface TaskDao {

    // Flow = "żywe zapytanie"
    // Emituje nową listę przy każdej zmianie tabeli
    @Query("SELECT * FROM tasks
            ORDER BY created_at DESC")
    fun getAllTasks():
        Flow<List<TaskEntity>>

    // suspend: operacja IO
    // Nie wolno wywołać na wątku UI!
    @Insert(onConflict =
        OnConflictStrategy.REPLACE)
    suspend fun upsert(task: TaskEntity)

    @Query("UPDATE tasks
            SET is_done = :done
            WHERE id = :id")
    suspend fun setDone(
        id: String, done: Boolean
    )

    @Query("DELETE FROM tasks WHERE id = :id")
    suspend fun delete(id: String)
}
Linia koduCo oznacza i dlaczego
interface TaskDao Room generuje impl — piszesz tylko interfejs z adnotacjami. Room (przez KSP) generuje klasę TaskDao_Impl w czasie kompilacji. Analogia: jak Retrofit dla HTTP.
Flow<List<TaskEntity>> Reaktywne zapytanie — po insert/update/delete Room automatycznie ponownie wykonuje zapytanie i emituje nową listę. Brak ręcznego odświeżania.
OnConflictStrategy.REPLACE Upsert — jeśli rekord z tym id istnieje → zastąp. Nowy → wstaw. Jedna operacja zamiast insert+update. Niezbędne przy Offline-First.
suspend fun delete Zawsze suspend — operacje bazodanowe są I/O. Muszą być w coroutine. Room sprawdza to w czasie kompilacji — błąd jeśli wywołasz na Main wątku.
PokemonApiService.kt 
// interface: Retrofit generuje implementację
// przez Dynamic Proxy w runtime
interface PokemonApiService {

    // GET https://pokeapi.co/api/v2/pokemon
    //        ?limit=20&offset=0
    @GET("pokemon")
    suspend fun getPokemonList(
        @Query("limit")  limit:  Int = 20,
        @Query("offset") offset: Int = 0,
    ): PokemonListDto

    // GET https://pokeapi.co/api/v2/pokemon/pikachu
    @GET("pokemon/{name}")
    suspend fun getPokemonDetail(
        @Path("name") name: String
    ): PokemonDetailDto
}

// DTO — odzwierciedla JSON 1:1
data class PokemonDetailDto(
    val id:   Int,
    val name: String,

    // podkreślnik w JSON → @SerializedName!
    @SerializedName("base_experience")
    val baseExperience: Int?,

    val height:  Int,   // decymetry
    val weight:  Int,   // hektogramy
)
Linia koduCo oznacza i dlaczego
@GET("pokemon") Metoda + ścieżka — Retrofit łączy z baseUrl: "https://.../api/v2/" + "pokemon" = kompletny URL. Uwaga: baseUrl MUSI kończyć się "/"!
@Query("limit") Query string — dołącza parametr do URL: ?limit=20. Wartość null → parametr jest pomijany. Domyślna wartość = parametr opcjonalny.
@Path("name") Path parameter — wstawia wartość do {name} w URL. "pokemon/{name}" + "pikachu" → "pokemon/pikachu". Automatyczny URL-encoding.
@SerializedName("base_experience") Mapowanie nazw — JSON ma "base_experience" (snake_case), Kotlin ma baseExperience (camelCase). Bez tej adnotacji Gson nie znajdzie pola → null!
val baseExperience: Int? Nullable = może być null w JSON — niektóre Pokémony nie mają base_experience. Brak "?" → JsonSyntaxException przy parsowaniu null.
RetrofitClient.kt 
// object = singleton w Kotlinie
// Tworzenie OkHttp/Retrofit jest kosztowne
// → inicjalizuj raz, reużywaj wszędzie
object RetrofitClient {

    private const val BASE_URL =
        "https://pokeapi.co/api/v2/"

    private val logging =
        HttpLoggingInterceptor().apply {
            // debugImpl only — nie loguj w release!
            level = Level.BODY
        }

    private val okHttp =
        OkHttpClient.Builder()
            .addInterceptor(logging)
            .connectTimeout(30, SECONDS)
            .readTimeout(30, SECONDS)
            .build()

    val api: PokemonApiService by lazy {
        Retrofit.Builder()
            .baseUrl(BASE_URL)
            .client(okHttp)
            .addConverterFactory(
                GsonConverterFactory.create()
            )
            .build()
            .create(PokemonApiService::class.java)
    }
}
Linia koduCo oznacza i dlaczego
object RetrofitClient Singleton — jedna instancja na całą aplikację. OkHttp utrzymuje pulę połączeń TCP — wiele instancji = wiele pul = marnowanie zasobów.
BASE_URL kończy się "/" Wymagane! — bez końcowego "/" Retrofit rzuca IllegalArgumentException. baseUrl + ścieżka @GET muszą się prawidłowo złożyć.
Level.BODY Pełny log HTTP — loguje nagłówki + ciało żądania i odpowiedzi do Logcat. Tylko debugImplementation — ciało może zawierać dane użytkownika!
by lazy { } Leniwa inicjalizacja — api tworzone przy pierwszym użyciu, nie przy imporcie. .create() generuje implementację interfejsu przez Dynamic Proxy.
Result.kt 
// sealed class: kompilator zna WSZYSTKIE podklasy
// when(result) MUSI obsłużyć każdą → bezpieczne
sealed class Result<out T> {

    data class Success<T>(
        val data: T
    ) : Result<T>()

    data class Error(
        val message: String,
        val cause: Throwable? = null,
        val code: Int? = null,
    ) : Result<Nothing>()

    data object Loading : Result<Nothing>()
}

// safeApiCall: każde wywołanie API opakowuje w try-catch
// Wynik: zawsze Result<T>, nigdy rzucony wyjątek
suspend fun <T> safeApiCall(
    call: suspend () -> T
): Result<T> = try {
    Result.Success(call())
} catch (e: HttpException) {
    val msg = when (e.code()) {
        404 -> "Nie znaleziono zasobu"
        401 -> "Wymagane logowanie"
        429 -> "Zbyt wiele żądań"
        in 500..599 -> "Błąd serwera"
        else -> "Błąd HTTP ${e.code()}"
    }
    Result.Error(msg, e, e.code())
} catch (e: UnknownHostException) {
    Result.Error("Brak połączenia z internetem", e)
} catch (e: SocketTimeoutException) {
    Result.Error("Przekroczono czas połączenia", e)
} catch (e: IOException) {
    Result.Error("Błąd sieci", e)
}
Linia koduCo oznacza i dlaczego
sealed class Result<out T> out T = kowariantność — Result<Dog> może być użyty jako Result<Animal>. Dzięki temu Error i Loading mogą mieć typ Result<Nothing> i pasować do Result<T>.
Result<Nothing> Nothing = podtyp wszystkiego — Result.Error(…) pasuje zarówno do Result<List<Pokemon>> jak i Result<Pokemon>. Jeden Error dla wszystkich typów.
suspend () -> T Lambda suspendująca — safeApiCall przyjmuje dowolne wywołanie API jako lambdę. Generyczne <T> = działa dla List<Pokemon>, Pokemon, Unit…
HttpException vs UnknownHostException Różne błędy, różne komunikaty — HTTP 404 to "nie ma takiego zasobu", UnknownHostException to "brak internetu". Użytkownik widzi odpowiedni komunikat.
in 500..599 Range w when — Kotlin pozwala użyć zakresu jako warunku. Wszystkie kody 5xx (błąd serwera) obsługiwane jedną linią.
PokemonViewModel.kt — obsługa Result 
fun loadPokemons() {
    viewModelScope.launch {
        // 1. Natychmiast pokaż spinner
        _uiState.update { it.copy(isLoading = true) }

        // 2. Wywołaj API przez safeApiCall
        val result = safeApiCall {
            api.getPokemonList()
        }

        // 3. when() musi obsłużyć KAŻDY stan
        // Pominięcie → błąd kompilacji, nie crash!
        _uiState.update { state ->
            when (result) {
                is Result.Success ->
                    state.copy(
                        isLoading = false,
                        pokemons  = result.data
                            .results
                            .map { it.toDomainModel() },
                        error     = null
                    )
                is Result.Error ->
                    state.copy(
                        isLoading = false,
                        error     = result.message
                    )
                Result.Loading ->
                    state.copy(isLoading = true)
            }
        }
    }
}

fun retry() { loadPokemons() }
Linia koduCo oznacza i dlaczego
_uiState.update { it.copy(...) } Atomowa aktualizacja — update() gwarantuje że nie zgubisz zmian przy równoległym dostępie. copy() zostawia niezmienione pola niezmienione.
is Result.Success Smart cast — po sprawdzeniu "is Result.Success" kompilator wie że result to Success i pozwala użyć result.data bez castowania.
result.data.results.map { ... } DTO → Domain Model — API zwraca DTO (Data Transfer Object). toDomainModel() konwertuje na Domain Model. ViewModel i UI nigdy nie widzą DTO.
fun retry() { loadPokemons() } Przycisk Retry — UI woła onRetry, ViewModel woła loadPokemons(). Logika powtórzenia w jednym miejscu — UI nie musi o niej wiedzieć.
Słownik pojęć

Kluczowe koncepty kursu

Sześć pojęć które pojawiają się w każdym ćwiczeniu. Jeśli rozumiesz każde z nich — rozumiesz fundamenty profesjonalnego kodu Android.

01 — ARCHITEKTURA

MVVM — podział odpowiedzialności

View (Compose) → tylko wyświetla stan.
ViewModel → logika, stan, akcje.
Model (Room/API) → dostarcza dane.

Separacja sprawia że możesz testować ViewModel bez emulatora i zmieniać UI bez ruszania logiki.

02 — REAKTYWNOŚĆ

StateFlow i rekomposycja

StateFlow emituje nowy obiekt UiState przy każdej zmianie. Compose wywołuje ponownie tylko te @Composable które odczytują zmienione pole. Zero ręcznego odświeżania UI.

03 — ASYNCHRONICZNOŚĆ

Coroutines i suspend fun

suspend fun zawiesza się czekając na I/O bez blokowania wątku UI. viewModelScope.launch tworzy coroutine auto-anulowaną przy zniszczeniu ViewModel. Brak ANR, brak memory leaks.

04 — BEZPIECZEŃSTWO

Null Safety Kotlina

String ≠ String?. Kompilator wymusza sprawdzenie null zanim użyjesz wartości nullable. NullPointerException w Kotlin jest trudne do napisania przez przypadek — to eliminuje całą klasę bugów.

05 — BEZPIECZEŃSTWO TYPÓW

Sealed class i when

Sealed class definiuje zamkniętą hierarchię stanów — tylko te podklasy które zadeklarujesz. when(result) musi obsłużyć każdą z nich. Pominięcie → błąd kompilacji, nie crash.

06 — DANE

Single Source of Truth

UI zawsze czyta z Room, nigdy bezpośrednio z API. Sieć aktualizuje Room → Room Flow emituje nowe dane → UI się odświeża. Jeden kierunek przepływu = brak niespójności, działanie offline.

FAQ

Najczęstsze pytania studentów

Pytania które pojawiają się podczas każdej edycji kursu PAM.

Muszę znać Javę żeby przystąpić do kursu?+
Nie. Kotlin to oficjalny język Androida i możesz zacząć bezpośrednio od niego. Jest nowocześniejszy i bezpieczniejszy niż Java. Wystarczy podstawowa znajomość programowania obiektowego w dowolnym języku (C++, Python, Java, C#). Jeśli znasz Javę — przyspieszy to naukę Kotlina, ale nie jest wymagane.
Czym Jetpack Compose różni się od starego systemu XML?+
Stary system: opisujesz UI w XML, potem ręcznie aktualizujesz widoki w Kotlin przez findViewById() i settery. Łatwo o niespójność — widok i stan mogą się rozjechać. Compose: opisujesz UI jako funkcje Kotlin i podajesz stan. Gdy stan się zmieni, Compose sam decyduje co przerysować — optymalnie i bez błędów. Stary system View jest de facto zamrożony przez Google, Compose to przyszłość Androida.
Dlaczego operacje bazodanowe i sieciowe muszą być suspend?+
Aplikacja Android ma jeden wątek UI — odpowiada za rysowanie ekranu (60 fps) i obsługę dotyku. Jeśli wywołasz operację sieciową bezpośrednio na tym wątku, ekran zamrozi się na czas oczekiwania — system może ubić aplikację (ANR). suspend fun zawiesza się czekając na wynik bez blokowania wątku — kod wygląda jak synchroniczny, ale działa asynchronicznie. Wywołanie na Main wątku bez suspend → NetworkOnMainThreadException.
Co to jest DTO i po co oddzielać od Domain Model?+
DTO (Data Transfer Object) odzwierciedla strukturę JSON 1:1 — ma @SerializedName, dokładnie te same pola co JSON. Domain Model to czyste dane które widzi ViewModel i UI — przeliczone jednostki, sformatowane teksty, bez szczegółów API. Gdy API zmieni nazwę pola "base_experience" na "base_exp" — zmienisz @SerializedName tylko w DTO. ViewModel, UI, testy — zero zmian. Separacja = odporność na zmiany API.
Dlaczego baseUrl w RetrofitClient MUSI kończyć się "/"?+
Retrofit łączy baseUrl z ścieżką @GET przez konkatenację. Bez końcowego ukośnika: "https://api.example.com/v2" + "pokemon" → "https://api.example.com/v2pokemon" (błędny URL). Z ukośnikiem: "https://api.example.com/v2/" + "pokemon" → "https://api.example.com/v2/pokemon" (poprawny). Brak "/" → IllegalArgumentException przy budowaniu Retrofit.
Czym różni się Flow.collect() od LiveData.observe()?+
LiveData był Androidowym rozwiązaniem problemu reaktywności. Flow to część Kotlin Coroutines — działa na każdej platformie, nie tylko Android. Flow ma bogatszy zestaw operatorów (map, filter, combine, flatMapLatest…), obsługuje backpressure i cold/hot streams. Różnica w użyciu: LiveData automatycznie uwzględnia lifecycle. Flow wymaga collectAsStateWithLifecycle() lub viewModelScope.launch — ale daje więcej kontroli. Google rekomenduje Flow dla nowych projektów.
Jak działają @Entity migracje — co się stanie jak zmienię klasę bez migracji?+
Room oblicza hash schematu bazy danych z adnotacji @Entity. Przy uruchomieniu porównuje hash zapisany w bazie z hashem skompilowanym. Jeśli się różnią i nie ma migracji → crash z IllegalStateException: Room cannot verify the data integrity. Dlatego każda zmiana @Entity (nowe pole, zmiana nazwy, usunięcie) wymaga: (1) inkrementacji version w @Database, (2) dostarczenia obiektu Migration z SQL ALTER TABLE. Alternatywa w developmencie: fallbackToDestructiveMigration() kasuje bazę — nigdy w produkcji!
